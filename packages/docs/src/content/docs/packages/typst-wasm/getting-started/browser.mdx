title: Browser without Vite
description: Compile Typst documents in a browser without Vite.
---

If you are not using Vite, publish the browser-facing JavaScript, worker, engine
WASM, and font files from your own origin. Do not point a production page at
`node_modules`: package managers do not provide a stable public URL layout, and
the compiler entrypoint has dependencies that a browser cannot resolve by
itself.

## A mock public directory

The following is a layout for a static site. The names and URLs are examples;
map them to whatever directory your server exposes as `/`.

```text
public/
├── index.html
├── app.js
└── vendor/
    ├── typst-wasm/
    │   ├── index.js
    │   ├── worker/
    │   │   ├── browser.js
    │   │   └── web-worker.js
    │   └── engine/
    │       ├── engine.core.wasm
    │       ├── engine.core2.wasm
    │       └── engine.core3.wasm
    └── typst-fonts/
        ├── LibertinusSerif-Regular.otf
        └── NewCMMath-Regular.otf
```

For example, `public/vendor/typst-wasm/engine/engine.core.wasm` is served at
`/vendor/typst-wasm/engine/engine.core.wasm`. Relative URLs are preferable when
the site may be deployed below a path prefix.

## Install and prepare the assets

Install the runtime and the fonts as normal dependencies, and use a small
build script to copy the binary assets and bundle the JavaScript entrypoints:

```sh
npm install typst-wasm @typst-wasm/fonts
npm install --save-dev esbuild
```

A script such as `scripts/prepare-typst-assets.mjs` can create the `vendor`
directory on every build:

```js
import { cp, mkdir, rm } from "node:fs/promises";
import { fileURLToPath } from "node:url";
import { build } from "esbuild";

const root = new URL("../public/vendor/", import.meta.url);
const wasm = new URL("typst-wasm/", root);
const fonts = new URL("typst-fonts/", root);

await Promise.all([
  rm(wasm, { recursive: true, force: true }),
  rm(fonts, { recursive: true, force: true }),
]);
await Promise.all([
  mkdir(wasm, { recursive: true }),
  mkdir(fonts, { recursive: true }),
]);

// Bundling avoids exposing node_modules and resolves typst-wasm's dependencies.
await build({
  entryPoints: {
    index: "typst-wasm",
    "worker/browser": "typst-wasm/worker/browser",
    "worker/web-worker": "typst-wasm/worker/web-worker",
  },
  outdir: fileURLToPath(wasm),
  bundle: true,
  format: "esm",
  platform: "browser",
  target: "esnext",
});

const copy = async (specifier, destination) => {
  await cp(
    fileURLToPath(import.meta.resolve(specifier)),
    fileURLToPath(new URL(destination, root)),
  );
};

await Promise.all([
  copy(
    "typst-wasm/engine/engine.core.wasm",
    "typst-wasm/engine/engine.core.wasm",
  ),
  copy(
    "typst-wasm/engine/engine.core2.wasm",
    "typst-wasm/engine/engine.core2.wasm",
  ),
  copy(
    "typst-wasm/engine/engine.core3.wasm",
    "typst-wasm/engine/engine.core3.wasm",
  ),
  copy(
    "@typst-wasm/fonts/LibertinusSerif-Regular.otf",
    "typst-fonts/LibertinusSerif-Regular.otf",
  ),
  copy(
    "@typst-wasm/fonts/NewCMMath-Regular.otf",
    "typst-fonts/NewCMMath-Regular.otf",
  ),
]);
```

Run this script before your static-site build (for example,
`"prepare:typst-assets": "node scripts/prepare-typst-assets.mjs"`). Commit
your lockfile and run the preparation step in CI. Re-run it whenever either
package changes. This keeps the public directory intentionally small and
makes dependency updates reviewable; do not deploy the whole `node_modules`
directory.

## Browser code

`public/index.html` can load an ES module directly from a static server:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8" />
    <meta name="viewport" content="width=device-width" />
    <title>Typst in a plain browser page</title>
  </head>
  <body>
    <main>
      <h1>Typst output</h1>
      <div id="output"></div>
    </main>
    <script type="module" src="./app.js"></script>
  </body>
</html>
```

`public/app.js`:

```js
import { createTypstCompiler } from "./vendor/typst-wasm/index.js";
import { createWebWorker } from "./vendor/typst-wasm/worker/browser.js";

const asset = (path) => new URL(`./vendor/typst-wasm/${path}`, import.meta.url);
const font = (name) => new URL(`./vendor/typst-fonts/${name}`, import.meta.url);
const response = async (url) => {
  const result = await fetch(url);
  if (!result.ok) throw new Error(`Could not fetch ${url}: ${result.status}`);
  return result;
};

const compiler = await createTypstCompiler({
  backend: "auto",
  worker: () => createWebWorker(asset("worker/web-worker.js")),
  coreModules: {
    "engine.core.wasm": WebAssembly.compileStreaming(
      response(asset("engine/engine.core.wasm")),
    ),
    "engine.core2.wasm": WebAssembly.compileStreaming(
      response(asset("engine/engine.core2.wasm")),
    ),
    "engine.core3.wasm": WebAssembly.compileStreaming(
      response(asset("engine/engine.core3.wasm")),
    ),
  },
});

await compiler.addFonts(
  ...["LibertinusSerif-Regular.otf", "NewCMMath-Regular.otf"].map(
    async (name) =>
      new Uint8Array(await (await response(font(name))).arrayBuffer()),
  ),
);

await compiler.addSource("main.typ", "= Hello from a plain browser page");
const result = await compiler.compile({ main: "main.typ", format: "html" });
document.querySelector("#output").innerHTML = result.output;

addEventListener("pagehide", () => void compiler.dispose(), { once: true });
```

The example uses the same `createTypstCompiler` API as the Vite guide; only the
asset URLs differ. The compiler also supports `pdf`, `png`, `svg`, and `bundle`
output.

## Serve and deploy

Do not open the page with `file://`; use an HTTP server. Serve `.wasm` files as
`application/wasm`, and keep the page, worker, JavaScript, WASM, and fonts on
the same origin. The worker backend also requires cross-origin isolation:

- `Cross-Origin-Opener-Policy: same-origin`
- `Cross-Origin-Embedder-Policy: require-corp`

Verify `crossOriginIsolated` in the browser. See the
[browser deployment requirements](../../deployment/browser-requirements/) for
complete hosting guidance. If vendor files use immutable caching, include a
package version or content hash in their public path when replacing them.
