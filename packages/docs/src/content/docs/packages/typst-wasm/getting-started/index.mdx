Here's what it looks like to initialise `typst-wasm`:

```ts
import { createTypstCompiler } from "typst-wasm";
import { createWebWorker } from "typst-wasm/worker/browser";
import workerUrl from "typst-wasm/worker/web-worker?worker&url";
import coreUrl from "typst-wasm/engine/engine.core.wasm?url";
import core2Url from "typst-wasm/engine/engine.core2.wasm?url";
import core3Url from "typst-wasm/engine/engine.core3.wasm?url";

const compiler = await createTypstCompiler({
  backend: "auto",
  worker: () => createWebWorker(workerUrl),
  coreModules: {
    "engine.core.wasm": WebAssembly.compileStreaming(fetch(coreUrl)),
    "engine.core2.wasm": WebAssembly.compileStreaming(fetch(core2Url)),
    "engine.core3.wasm": WebAssembly.compileStreaming(fetch(core3Url)),
  },
});
```

A lot of imports, yes. The benefit of this API is that you get to decide where the assets come from. This instantiation scheme is why typst-wasm has such broad compatibility.

Lets go through line-by-line to understand what is happening here.

## backend: "auto"

typst-wasm supports two asynchronous backends: JSPI and workers. `backend: "auto"` will choose the backend that is supported by the runtime you run it in. To test what backend will be chosen for a runtime, replace `createTypstCompiler` with `selectAutomaticBackendKind` in the above example and log out the result.

In cases where you want to opt out of runtime detection, you can pass `backend: "jspi"` or `backend: "worker"`.

## worker: () => createWebWorker(workerUrl)

Web workers and Node.js worker threads are not supported everywhere, and bundlers deal with workers in a variety of ways.

We let you supply a URL to the worker in a way that fits your bundler and deployment setup.

## coreModules

WASM imports are currently very experimental and as such can't be reliably abstracted away. To provide broad bundler and deployment setup support, you provide the compiled WASM modules.
