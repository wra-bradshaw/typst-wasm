`typst-wasm` provides two backends to help you compile Typst in any runtime:

- The "workers" backend compiles your Typst in a separate worker. This backend is supported in most runtimes that support web workers or worker threads.
- The "JSPI" backend compiles your Typst on the main thread, but it requires experimental JSPI (which has [limited availability](https://caniuse.com/wf-wasm-jspi)).

Between the two backends we can support all major browsers engines (Chrome, Firefox, Safari) and all major server side runtimes (Node, Deno, Bun, workerd).

Use the unified `typst-wasm` compiler API everywhere. Choose a host adapter only when configuring a worker: `typst-wasm/worker/node` for Node/Bun or `typst-wasm/worker/browser` for browsers.

<Aside type="tip" title="Using another framework?">
  React, Vue, Svelte, and similar frameworks use the browser/Vite or Node.js
  guide. The framework does not change the typst-wasm runtime.
</Aside>

## Runtime and deployment are separate

The runtime determines which import and compiler backend to use. Your hosting provider determines how those assets and headers are deployed. See the [deployment guides](../deployment/) after completing a runtime guide.

For browser compilation without Vite, see [Use the browser without Vite](../how-to/browser-without-vite/).
